문서나 스크린샷에서 테이블을 paddle 이나 vlm 으로 추출하는데 한계를 겪고있는데, 문서를 카메라로 찍은건 어느정도 잘 추출되는데, 라인이 정확하게 없는 테이블은 이게 삐뚫어진채로 찍거나 가로로 폰을 돌려서 세로로 찍거나 하면 읽지 못함. orientation등으로 수정은 어느정도 되는데, 삐둟어진건 행을 실수로 잘못 읽는 사례가 있어. 이걸 정확하게 읽기 위한 방법론 조사.기하학적 왜곡 보정과 딥러닝 기반 테이블 인식 파이프라인을 설계했습니다기하학적 왜곡 보정과 딥러닝 기반 테이블 인식 파이프라인을 설계했습니다테이블 추출 실패의 원인을 좀 더 세분화해서 보면, 사실 "라인 없는 표"와 "삐뚫어진 촬영"은 서로 다른 문제가 겹쳐 있어서 하나의 해법으로는 안 잡힙니다. 조사한 내용을 정리하면 아래와 같습니다.
1. 문제를 두 층위로 분리해야 함

기하학적 정규화(geometric normalization): orientation(0/90/180/270), skew(회전), perspective distortion(원근 왜곡) — 이건 OCR/VLM 이전에 이미지 자체를 펴는 문제
테이블 구조 복원(TSR, Table Structure Recognition): 라인이 없는 표에서 row/column을 어떻게 재구성하느냐

지금 겪는 "행을 잘못 읽는" 문제는 대부분 1번이 제대로 안 된 상태에서 2번을 시도하기 때문에 생깁니다. Row 셀 배치는 보통 y좌표 기반 클러스터링으로 이루어지는데, skew된 이미지에서는 y좌표 자체가 흐트러져서 클러스터링이 깨집니다.
2. "삐뚫어짐"은 대부분 in-plane rotation이 아니라 perspective distortion
폰카로 문서를 찍을 때 기울어지는 건 단순 회전(rotate)이 아니라 카메라와 문서 평면 사이의 사각(斜角) 촬영, 즉 투영 왜곡인 경우가 많습니다. 단순 각도 회전(Hough transform 기반 skew correction)으로는 이걸 완전히 못 폅니다. 이 방식은 스캐너로 스캔한 문서가 살짝 기운 경우에 최적화된 고전적 기법이라, 사진 촬영의 4점 원근왜곡에는 한계가 있습니다.
해법: 문서 경계(4개 꼭짓점)를 검출해서 homography(4-point perspective transform)로 평면화하는 게 먼저입니다. 고전적으로는 OpenCV contour 검출 + cv2.getPerspectiveTransform, 최근에는 딥러닝 기반 dewarping 모델들이 훨씬 강건합니다:

UVDoc (grid-based neural unwarping)
DocTr++, D2Dewarp — 최근 arXiv에 나온 모델들로, 표 이미지 전용 dewarping을 다루는 연구도 있습니다 U-Net 기반 모델로 표 이미지를 언워핑하는 방식이 제안되었고, 표 구조 자체의 핵심 요소를 세그멘테이션해서 왜곡을 잡아내는 접근이 MS-SSIM 기준 15% 개선을 보였습니다 Springer

가로로 찍어서 90도 돌아간 경우는 별도로 orientation classifier(경량 CNN, PP-LCNet류)로 먼저 0/90/180/270을 정규화하는 단계를 넣는 게 맞습니다. 이건 skew 보정과 별개 모듈이어야 합니다.
3. 라인 없는 표(borderless table) 구조 복원
라인 기반 방법(흰 여백 gap 탐지, contour 기반 셀 복원)은 애초에 skew에 취약합니다 — 수평/수직 gap을 찾는 방식이라 이미지가 살짝만 기울어도 gap이 사라지거나 잘못 잡힙니다. 클리어하고 검출 가능한 라인이 있어야 셀 식별이 제대로 되고, 라인이 없는 경우엔 흰 여백의 수직/수평 gap을 이미지 임계처리 후 직접 재구성해야 하는데 이 방식 자체가 회전에 민감합니다. Medium
더 강건한 대안:

Table Transformer (TATR, microsoft/table-transformer): DETR 기반으로 테이블 구조 인식 모델이 row, column, 그리고 병합/분할된 셀까지 포함해 구조를 예측하며, 라인이 없어도 상당히 견고하게 작동합니다. 다만 촬영 왜곡이 심하면 사전학습 분포에서 벗어나 정확도가 떨어질 수 있어 파인튜닝이 필요할 수 있습니다. GitHub

PaddleOCR-VL이 정확히 이 문제(skew + 회전 촬영)를 타겟으로 최근 업데이트되었습니다. PP-DocLayoutV3 알고리즘을 처음 도입해서 불규칙 형태 위치 지정을 다루는데, Skew(기울어짐), Warping(휘어짐), Scanning, Illumination, Screen Photography 5가지 어려운 시나리오를 정면으로 다룹니다. 2026년 1월 출시된 PaddleOCR-VL-1.5는 irregular-shaped bounding box localization을 지원해서 Skew, Warping, Screen Photography, Illumination, Scanning 같은 실제 환경에서 뛰어난 성능을 보이고, 2026년 5월 출시된 1.6 버전은 OmniDocBench v1.6에서 96.3% 정확도로 텍스트/수식/표 인식 전반에서 업계 최고 성능을 보입니다. 5. 실전 파이프라인 제안
1. 문서 경계 검출 (contour or DL 기반) → 4-point perspective transform으로 평면화
2. Orientation classifier (0/90/180/270) 로 회전 정규화
3. PaddleOCR-VL-1.6 (또는 PP-StructureV3) 로 표 인식
   - 실패/의심스러운 경우 Table Transformer로 cross-check
4. 후검증: 각 row의 셀 개수가 헤더 컬럼 수와 일치하는지 rule-based로 체크
   → 불일치 시 해당 row만 crop해서 재인식 (2차 pass)
핵심은 1~2단계(기하 정규화)를 표 인식 모델에만 맡기지 말고 명시적 전처리 단계로 분리하는 것입니다. 지금처럼 VLM/PaddleOCR에 왜곡된 이미지를 통째로 넣으면, 모델이 내부적으로 암묵적 보정을 시도하다 실패하는 게 row 오분류로 나타나는 걸로 보입니다.

반드시 감안해야 할 전제조건
이 숫자를 그대로 믿으면 안 되는 이유가 몇 가지 있습니다:

72 DPI PDF 페이지 기준입니다. 폰카로 찍은 표 사진은 보통 그보다 해상도가 훨씬 높습니다(수 MB짜리 이미지). 해상도가 올라가면 layout 분석과 VLM 인코딩 비용이 늘어나서 실제 처리량은 이 표보다 낮을 가능성이 큽니다. 실사용 이미지로 자체 벤치마크를 한 번 돌려보는 게 필요합니다.
"배치 512" 처리량은 처리량 최적화 모드입니다. 이건 대량 문서를 쌓아놓고 스루풋 위주로 돌릴 때 수치이고, 실시간 API처럼 요청이 들어오는 즉시 짧은 지연시간으로 응답해야 하는 서빙 방식(continuous batching, 낮은 max-num-seqs)에서는 처리량이 떨어집니다. "초당 100개"가 실시간 응답 SLA를 포함하는지, 아니면 배치로 몰아서 처리해도 되는지에 따라 필요한 GPU 수가 꽤 달라집니다.
VRAM이 이미 꽉 차 있습니다. A100 80GB 기준으로 62GB를 이미 쓰고 있어서, 동시성을 더 올리려면 GPU를 늘리는 것 외에는 여유가 별로 없습니다. H20은 77GB로 거의 한계치라 96GB 카드가 아니면 빡빡합니다.
여기엔 앞서 얘기한 dewarping/deskew 전처리 단계 비용이 안 들어가 있습니다. VLM 인퍼런스 앞단에 perspective correction, orientation classification을 넣으면 그만큼 파이프라인 전체 처리량은 더 떨어집니다. 이 전처리는 보통 CPU나 경량 모델로 처리하니 GPU 병목까지는 안 가지만, latency budget에는 포함해야 합니다.

결론적으로: A100 기준 최소 50대, 여유 포함 60~65대 규모의 클러스터를 잡는 게 현실적인 시작점입니다. 다만 정확한 숫자는 실제 촬영 이미지 해상도와 "100개/초"가 배치형인지 실시간형인지에 따라 꽤 갈리니, 소규모(4~8 GPU)로 실사용 이미지 기준 자체 벤치마크를 먼저 돌려보고 그 실측값으로 스케일업 계산을 다시 하는 걸 추천합니다.GCP 배포 옵션과 GPU 요구사항을 검토했다GCP 배포 옵션과 GPU 요구사항을 검토했다조사해보니 PaddleOCR-VL은 Vertex AI Model Garden에 완전관리형으로 올라가 있지 않습니다. 즉 GCP에서 쓰려면 직접 배포(self-host)해야 합니다. 방법은 세 가지로 나뉩니다.
배포 옵션 3가지
1) Cloud Run GPU (서버리스, 가장 가벼운 시작점)
실제로 커뮤니티 피드백에 "Google Cloud Run GPU 최소 구성(L4 GPU + 16GB RAM)에 완벽하게 들어간다"는 사례가 있습니다 공식 Docker 이미지를 그대로 Cloud Run에 올리면 됩니다. 트래픽이 없을 때 0으로 스케일 다운되니 프로토타입/저부하 검증 단계에는 이게 제일 편합니다. 다만 지난번 계산했던 초당 100개 수준의 지속 부하에는 GPU 1장짜리 서버리스로는 부족합니다.
2) Compute Engine GPU VM (단순 자체 서버)
Docker 이미지나 pip install "paddleocr[doc-parser]" + vLLM/FastDeploy 백엔드로 VM에 직접 올리는 방식. GPU 1~수 장 규모 테스트에 적합합니다.
3) GKE (GPU 노드풀, 스케일 필요할 때)
지난번 계산한 것처럼 A100 50~65장 규모가 필요하다면 이 방식이 맞습니다. GKE는 H200/H100/A100/L4 GPU 하드웨어에서 오토스케일링, 노드풀 관리를 지원하고, Vertex AI 대비 인프라를 직접 컨트롤할 수 있습니다 vLLM 서빙 컨테이너를 GKE에 배포하는 공식 가이드 패턴(Gemma 서빙 예시와 동일한 구조)을 그대로 PaddleOCR-VL 컨테이너에 적용하면 됩니다.
GPU 선택 시 주의점
PaddleOCR-VL 공식 문서에 따르면 vLLM/FastDeploy/SGLang 백엔드는 Compute Capability(CC) ≥ 8 하드웨어를 요구합니다. T4/V100(CC 7.x)은 vLLM에서 시작은 되지만 timeout이나 OOM이 잦아 비권장입니다.
GCP에서 CC≥8을 만족하는 GPU 인스턴스 계열:

g2 (L4, CC 8.9) — 저비용, Cloud Run GPU 최소 구성에도 쓰인 바로 그 GPU
a2 (A100, CC 8.0) — 지난번 처리량 계산의 기준 GPU
a3 (H100, CC 9.0) — 가장 높은 처리량, 계산상 A100보다 적은 대수로 목표 달성 가능

(이중화(zonal redundancy)를 켜면 GPU가 $0.0002909/초로 올라가서 합계가 더 비싸집니다.)
L4는 최소 4vCPU + 16GiB 메모리가 강제 조건이라 GPU만 떼서 계산하면 안 되고 항상 이 조합으로 계산해야 합니다.
중요한 함정: GPU 붙으면 "요청 처리 중"이 아니라 "인스턴스 살아있는 동안" 전부 과금
일반 Cloud Run은 기본적으로 요청을 처리하는 동안만 과금(request-based billing)되지만, GPU 서비스는 인스턴스 생존 시간 전체를 초 단위로 과금합니다 즉 요청 사이에 인스턴스가 죽지 않고 대기하고 있으면, 그 대기 시간도 그대로 돈이 나갑니다.
"5분 간격으로 계속 있다"의 두 가지 해석
해석 A: 인스턴스가 죽지 않고 계속 떠 있는 상태 (5분마다 요청이 와서 idle timeout 전에 계속 살아남는 경우)
이 경우 사실상 24시간 상시 가동과 동일합니다.

시간당 약 $1.16
일 24시간 × $1.16 ≈ $27.9/일
월 730시간 기준 ≈ 약 $848/월

해석 B: 요청 처리 후 매번 0으로 스케일다운되고, 5분마다 새로 뜨는 경우
이러면 실제 처리 시간만 과금됩니다. 인스턴스 시작에 약 5초가 걸리니, 예를 들어 요청당 처리 시간이 15초(시작 5초 + 처리 10초)라고 가정하면:

하루에 5분 간격 → 288회 호출 (24h × 60 / 5)
회당 15초 → 하루 총 4,320초 (72분)
시간 비용 $1.16/hr 기준 → 72분 ≈ 1.2시간 × $1.16 ≈ $1.4/일, 월 약 $42/월

즉 스케일다운 여부에 따라 월 $42 vs $848로 20배 차이가 납니다. 어느 쪽이 실제 상황인지가 비용에 결정적입니다.
실무 포인트

5분 간격으로 요청이 오는데 Cloud Run이 기본 idle timeout 안에 다음 요청을 계속 받으면, 스케일다운이 발생하지 않고 사실상 상시 가동으로 청구됩니다. 진짜 비용을 줄이려면 인스턴스가 매번 확실히 0으로 내려가도록 max-instance-idle 같은 설정을 확인하거나, 아예 min-instances=0으로 두고 콜드스타트(약 5초)를 감내하는 게 낫습니다.
반대로 5분마다 콜드스타트 5초씩 감내하기 싫다면(레이턴시 민감), min-instances=1로 상시 유지해야 하고 그때는 월 $848 수준을 그냥 지불하는 게 맞습니다.

지금 상황이 "5분마다 새 요청이 오는데 그 사이엔 꺼져도 되는지" 아니면 "레이턴시 때문에 계속 켜둬야 하는지"에 따라 답이 완전히 달라지니, 어느 쪽인지 알려주시면 더 정확히 좁혀드릴 수 있습니다.
